# Finmo - Expense Tracker

Finmo is a robust, full-stack expense tracking application designed to provide users with a seamless and accurate way to log, manage, and review their financial data. It is built with a strong focus on data correctness, handling edge cases gracefully, and avoiding common financial software pitfalls such as floating-point precision errors.

## 🏗 Architecture

The project follows a standard Client-Server architecture:

*   **Frontend**: A modern Single Page Application (SPA) built with Next.js, React, and TypeScript. It communicates with the backend via RESTful APIs.
*   **Backend**: A high-performance, asynchronous RESTful API built with Python and FastAPI. It handles data validation, business logic, and database operations.
*   **Database**: MongoDB, a NoSQL database, used for storing expense records flexibly and efficiently.

## 💻 Tech Stack

### Frontend
*   **Next.js 14 (App Router)**: For routing and server-side rendering capabilities.
*   **React 18**: Core UI library.
*   **TypeScript**: For static typing and better developer experience.

### Backend
*   **FastAPI**: A modern, fast web framework for building APIs with Python based on standard Python type hints.
*   **Pydantic**: Data validation and settings management using Python type annotations.
*   **Motor**: Asynchronous Python driver for MongoDB, allowing non-blocking database operations.

### Database
*   **MongoDB**: Document-based NoSQL database.

## 🗄 Database Design

The database consists of an `expenses` collection with the following document structure:

| Field | Type | Description |
| :--- | :--- | :--- |
| `_id` | ObjectId | Unique identifier generated by MongoDB. |
| `amount` | String | The expense amount. Stored as a string in the DB to avoid floating-point precision loss, and parsed safely to Decimal in the backend. |
| `category` | String | The category of the expense (e.g., Food, Transport). |
| `description` | String | A brief description of the expense. |
| `date` | String | The date the expense occurred, stored in ISO format (YYYY-MM-DD) to ensure accurate string-based sorting. |
| `created_at` | DateTime | Timestamp of when the record was created. |
| `idempotency_key` | String | A unique token (UUID) provided by the client during creation. It has a unique index in the database to prevent duplicate insertions if a request is retried. |

## 🚀 Setup and Installation

### Prerequisites
*   Node.js (v18+ recommended)
*   Python 3.10+
*   MongoDB (running locally or a MongoDB Atlas cluster)

### 1. Backend Setup

1.  Navigate to the backend directory:
    ```bash
    cd backend
    ```
2.  Create a virtual environment:
    ```bash
    python -m venv .venv
    ```
3.  Activate the virtual environment:
    *   **Mac/Linux**: `source .venv/bin/activate`
    *   **Windows**: `.venv\Scripts\activate`
4.  Install the required dependencies:
    ```bash
    pip install -r requirements.txt
    ```
5.  Set up Environment Variables. Create a `.env` file in the `backend` directory based on `.env.example`:
    ```bash
    cp .env.example .env
    ```
    Update the `.env` file with your details:
    ```env
    MONGODB_URL=mongodb://localhost:27017
    DB_NAME=fenmo
    FRONTEND_ORIGIN=http://localhost:3000
    ```
6.  Start the FastAPI server:
    ```bash
    uvicorn main:app --reload
    ```
    The API will be available at `http://localhost:8000`. You can access the interactive API documentation at `http://localhost:8000/docs`.

### 2. Frontend Setup

1.  Navigate to the frontend directory:
    ```bash
    cd frontend
    ```
2.  Install the dependencies:
    ```bash
    npm install
    ```
3.  Start the development server:
    ```bash
    npm run dev
    ```
    The application will be available at `http://localhost:3000`.

## 💡 Key Technical Choices & Considerations

*   **Money Handling**: JavaScript floating-point errors (`0.1 + 0.2 = 0.30000000000000004`) are strictly avoided. The frontend implements a dedicated `parseAmount()` utility to handle edge cases, and the backend converts all amounts to Python `Decimal` objects to ensure 100% precision.
*   **Idempotency**: Network instability can cause clients to retry requests, potentially creating duplicate expense entries. The API solves this by using `idempotency_key`. The client sends a unique UUID with each creation request. The MongoDB database has a unique index on this field, ensuring that repeated requests simply return the existing record instead of creating a new one.
*   **Validation**: Both frontend and backend perform strict validation. The backend utilizes Pydantic to ensure amounts are strictly positive, strings are stripped of whitespace, and dates are valid.
